Webpack 究竟是什么

webpack is a nodule bundler

在模块化编程中，开发者将程序分解为功能离散的 chunk，并称之为 模块。 每个模块都拥有小于完整程序的体积，使得验证、调试及测试变得轻而易举。 精心编写的 __模块__提供了可靠的抽象和封装界限，使得应用程序中每个模块都具备了条理清晰的设计和明确的目的。

在 mode: 'development' 开发者模式下，sourcemap 默认已经被配置，当被关闭后，如果代码出错，会提示打包后的错误代码位置，而不是源代码的错误代码位置

需求：在代码错误的情况下，调试工具中不告诉打包后的代码哪里出错，而是指出源代码（src/index.js）的出错位置

如果不配置 sourcemap 会提示打包后的 dist 目录下 main.js 文件 96 行出错

source-map

sourceMap 它是一个映射关系，他知道 dist 目录下 main.js 文件 第 96 行，实际上对应的是 src 目录下 index.js 文件中的第一行

通过配置 sourcemap 就可以提示 index.js 文件第一行出错

此时打包内容会多出一个 main.js.map 文件，里面就是映射对应关系的描述（vlq 的编码集合）

inline-source-map

如果使用 inline-source-map 就不会有 main.js.map 文件，该文件会通过 data url 的方式直接写在 main.js 里面（变成了 base64 字符串放在最底部）

cheap-source-map

  • 提示行不提示列出错
  • 只负责业务代码错误，打包过程 loader 等出现错误不管

当我们代码量很大的时候，如果代码出错，而此时 devtool 配置的没有 cheap 前缀，sourcemap 会告诉你代码在第几行的第几个字符的地方出了问题，会精确到哪一行哪一列，但是这样的映射比较耗费性能，一般只希望告诉在哪一行出了问题就行

使用这种方式打包性能也会提升

cheap-module-source-map

  • 管业务代码与 loader 错误

使用 cheap 这种方式，此时 sourcemap 只针对业务代码，而不会管第三方模块的问题，如果要管第三方，则 cheap-module-source-map

eval

eval 是执行速度最快、性能最好的打包方式，也不会产生 main.js.map 文件，但是如果针对复杂代码，提示出的错误并不全面

小结

  • dev 环境下 sourcemap 推荐使用：eval-cheap-module-source-map（提示较全打包速度较快）

  • prod 环境下一般没必要有 sourcemap 的映射，但有时线上代码出现问题，希望也能用 sourcemap 定位问题，那么 sourcemap 推荐使用：cheap-module-source-map（提示效果会更好）

考点

sourcemap 原理，映射是怎样实现的，设计例如 eval-cheap-module-source-map 的思想
