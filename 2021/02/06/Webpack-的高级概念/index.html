Tree Shaking 的使用

问题说明：在打包后的 main.js 中同时打包了 add 与 minus 方法，这表明虽然 index.js 只引入了 add 方法，但是 把 math.js 里的内容都打包生成到了 main.js 里面，这是没有必要的，会使 main.js 文件变得很大，最理想的打包方式，引入什么就打包什么

解决方法：使用 Tree Shaking（只支持 ES Module 静态引入方式 — import … from …，commonjs 是动态引入方式）

使用方法：在 webpack.config.js 中做配置，development 模式默认没有 Tree Shaking 功能，需要加入 optimization 配置

同时要在 package.json 里面加入配置项 sideEffects，如 babel/poly-fill，不希望对其 tree shaking，就需要在 sideEffects 里面填写数组，"sideEffects": ["@babel/poly-fill"]，就对 babel/poly-fill 就不会有 tree shaking，"sideEffects": false代表都需要做 tree shaking，一般会对 css 文件不使用 tree shaking "sideEffects": [ "*.css" ]

再次打包后，main.js 显示如下，即使用了 tree shaking 也不会直接剔除，这是因为在开发环境下生成的代码需要做一些调试，如果 tree shaking 删除掉一些代码后再做调试，可能代码对应的行数（source-map）会混乱，但是如果将模式改为生产环境（此时 optimization 可以去掉，devtool 去掉 eval，sideEffects 还是需要写），tree shaking 就会完全生效

这种方式带来潜在的问题：假设 lodash 1.5Mb，业务逻辑代码 1Mb，打包后（假设不压缩）的代码为 2.5Mb 的 js 文件，这就意味着如果用户访问 index.html 文件，要想输出业务逻辑就需要先去加载一个 2.5Mb 的 js 文件

异步加载方式 index.js

同步代码比异步代码简介，为什么要写成异步代码，这是因为异步的写法可以实现懒加载行为

好处：借助 import 这种语法可以让页面加载速度更快

注：import 语句后面可以跟一个 then，说明 import 返回的是一个 promise 类型，这就意味着，如果想在代码里面使用 import 必要要使用 polyfill，例如 babel-polyfill，因为在低版本浏览器下很有可能不支持 promise 语法（新版本 babel-preset 里面内置了 babel-polyfill）

在 ES7 中标准中，引入了异步函数的概念，用异步函数就可以省略掉 promise 比较复杂的写法，变成简单的写法

假设引入 lodash 库，是否应该做代码分割：首先 chunks: ‘all’ 满足，它大于 20 kb 也满足，但是对于 minChunks: 2 这项，假设整个项目打包过后在 dist 目录下生成很多个 chunk 文件，如果有两个以上的文件里面需要依赖 lodash，那么就需要对 lodash 进行 code spliting，如果整个代码打包生成之后只有一个 chunk 用到了 lodash 库，就不满足minChunks: 2这项，就不会被代码分割

注：其实 optimization 除了 chunks: 'all'，其余用默认项就可以

实际应用场景：登录框，在首页加载的时候就不应该一起加载出来，只有点击登录框才去加载，但是问题又来了，这样做会不会使登录框显示太慢，这就利用到了 Preloading、Prefetching

与 prefetch 指令相比，preload 指令有许多不同之处：

  • preload chunk 会在父 chunk 加载时，以并行方式开始加载。prefetch chunk 会在父 chunk 加载结束后开始加载。
  • preload chunk 具有中等优先级，并立即下载。prefetch chunk 在浏览器闲置时下载。
  • preload chunk 会在父 chunk 中立即请求，用于当下时刻。prefetch chunk 会用于未来的某个时刻。
  • 浏览器支持程度不同。

因此建议 prefetch，等待核心业务代码加载完成之后再去加载

注：缓存不是最重要的点，应该把注意力转移到 code coverage 上

小知识点 chunkFilename

经常会在 webpack.common.js 中的 output 出现 chunkFilename

首先 index.js

当配置了 chunkFilename 打包后会生成一个 lodash.chunk.js 文件

打包后的 main.js 里面引入了 lodash，lodash 又被单独打包生成了一个文件，打包生成的文件在整个代码运行过程中，main.js 先执行，然后异步的加载 lodash.js，因此它并不是一个入口的 js 文件，是一个被 main.js 异步加载的间接的 js 文件，打包生成间接的 js 文件，就会走 chunkFilename 配置项

或者简单说，在打包后的 index.html 只引入了 main.js，并没有引入 lodash，是通过 main.js 再去引入进来的，这种间接引用的文件在打包的时候就会走 chunkFilename 配置项
