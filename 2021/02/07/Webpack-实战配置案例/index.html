需求：上面 string.js 代码中是通过自己写的字符串拼接，但是想要在库里面使用 lodash 中的字符串拼接方法，如下

此时重新打包生成的代码变成了 78kb，因为库中多引入了一个 lodash 库

问题：

他在使用库的同时也可能使用了 lodash，但是我们的 library 里面已经打包了 lodash，而现在用户的业务代码里面又打包了一次 lodash，所以在最终用户的代码里面就很容易存在两份 lodash 的代码

解决方法：在我们库的 webpack 配置文件中配置 externals

再次打包后，library.js 从 78kb 减少到了 1.6kb，这样配置 externals 打包意思是：打包过程中如果遇到了 lodash 库，就忽略这个库不打包到代码里面，但是此时别人无法直接使用我们的这个库，因为我们这个库里要用到 lodash，但是自己又没有 lodash，所以在用我的库的时候，要在业务代码最上面引入 lodash 库

最终我们提供给别人使用的是打包好的 library.js，要方便他人使用还要在 package.json 中做一些配置

发布

1.在 npm 官方网站上注册账号并登录：https://www.npmjs.com/

2.命令行中：npm adduser（添加用户名和密码）

3.添加用户名密码成功后，运行命令：npm publish（直接把项目发布到 npm 仓库上面，会报错，因为 library 会重名，写个独特的名字就行）

4.发布成功后，别人直接命令行：npm install library 就能够使用打包好的库

注：每个库的名字在 npm 仓库中是不能相同的

package.json 中增加一个 start 运行命令，表示打包后 dist 目录下的代码在一台服务器中启动

执行该命令后，在命令行中会显示已经启动服务器，如下所示

然后随便找一个进去，html 正确运行，当关闭该服务器后，在浏览器中刷新页面再次访问该域名，直接报服务器挂掉了，这就是一个传统网页方式

PWA 可以实现访问一个网站，第一次访问成功，但是突然之间服务器挂掉，第二次再重新访问网站的时候，可以在本地有一份缓存，可以直接用这个缓存将页面直接展示出来，因此即使服务器挂掉依然能够正常访问

实现

我们在写一些项目的时候经常要去发一些 ajax 请求

安装 axios：npm install axios --save-dev

index.js

此时是在 localhost 下请求另外一个域名的接口，正常应该是跨域，但是请求却成功了，这是因为对方服务器允许做跨域请求，所以能够正常拿到结果

在我们做开发的时候，前端的代码有可能是在开发环境下运行的，也有可能是在线上环境运行，两者请求的接口有可能是不一样的，所以如果在请求接口中将域名写死，那么请求的永远都是线上的服务器，所以一般做 ajax 请求接口时不会用绝对路径形式，而是会采用相对路径的形式

但是现在无法请求，需要在 webpack 配置文件中做配置（proxy 官方配置说明）

重新 run start 一下，请求成功，此时浏览器控制台 netword 下的 header.json 文件显示的请求的是 localhost:8080/react/api/header.json下的数据，正常情况下，这个数据是不存在的，之所以有数据返回就是因为配置了 proxy 代理，有了这个配置，当我们在代码里请求 /react/api/ 开头的接口时，proxy 就会将其代理到 http://www.dell-lee.com

通过这种 proxy 再做开发环境的联调就比较简单了

在实际开发中有可能遇到的场景：假设后端 header.json 端口暂时没法使用，但是可以先用一下 demo.json 的假数据测试接口，所以对于前端来说 header.json 是最终要请求的接口，但目前需要临时使用 demo.json

问题：对于上述场景不能直接在 axios 直接修改相对路径的请求地址，这样一旦忘记修改直接上线就会出现问题

解决：把它始终写成 header.json，不变动，在 webpack 配置中可以完成 header 与 demo 的转换

重新 run start 一下，请求成功，实现完毕，当后端 header.json 写好了直接将 'header.json': 'demo.json'注释掉

有时候转发的网址是 https，这时候还要做一步配置

注意：这里的 proxy 是 devServer 的 proxy 只有在开发环境才生效，因为只有在开发环境下才会用 webpackdevServer，线上不存在 webpackdevServer
